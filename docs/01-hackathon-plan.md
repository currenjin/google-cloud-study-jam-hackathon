## 120분 타임박스

### 0–30분: 뼈대 (목업 시드·씬 화면 + 실제 한 사이클)

- AI Studio에서 앱 바이브코딩 시작 (필요시 "Export to Antigravity")
- 먼저 목업으로 **시드 → 씬** UI를 완성한 뒤, 실제 **시드 입력 → Gemini 각본 → 이미지 1장**을 연결 (릴 재생 UI는 65–90분에)
- 프롬프트는 [03-prompts.md](03-prompts.md)에서 복붙
- 첫 사이클 돌면 즉시 git push

### 30–65분: 핵심 기능 (릴레이 + 일관성)

- Story Bible 생성 로직: 첫 시드에서 JSON 바이블 생성, 이후 모든 호출에 컨텍스트로 주입
- 턴 릴레이 UI: "다음 전개 한 줄" 입력 → 씬 생성 → "다음 사람에게 넘기기" (핫시트 방식, 계정/실시간 동기화 없음)
- 턴마다 `updated_story_summary` 검증 후 갱신
- 이미지는 9:16 고정, visual_style 프리픽스 강제
- localStorage에는 텍스트 상태만 저장하고 생성 이미지·API 키는 저장하지 않음

### 65–90분: 완성 릴 + 폴리시

- 릴 재생 화면: 지금까지의 컷을 세로 릴로 자동 재생 (컷 + 자막/대사 오버레이)
- 여유 있으면: Veo "시네마틱 모드" 버튼 추가 — 단, **데모 영상용 클립은 이 시간에 미리 생성 시작** (클립당 1–3분 걸림)
- 여유 없으면 Veo는 버린다. 이미지 릴만으로 데모 성립
- `VITE_DEMO_MODE=true` 전체 흐름과 강제 이미지 실패 → 텍스트 카드 폴백 확인
- `npm run build` 통과 확인

### 90–120분: 제출물 (절대 침범 금지)

- [ ] 데모 시나리오대로 화면 녹화 (03-prompts.md의 시드/전개 사용) — DEMO_MODE 아닌 실제 생성 화면으로만
- [ ] 데모 영상 업로드 (YouTube unlisted 또는 Google Drive) → 링크 확보
- [ ] README 작성: 프로젝트 소개, **실제로 사용한** 아키텍처(Vite + Gemini 각본/이미지 + Story Bible), 실행 방법, 스크린샷. Cloud Run은 실제 사용했을 때만 기재
- [ ] 최종 git push → repo가 public인지 확인
- [ ] Project Description 제출 양식 작성
- [ ] **로컬/GitHub에 전부 백업** — @gcplab.me 계정과 GCP 프로젝트는 행사 끝나면 즉시 삭제됨

## 스코프 규칙 (현장에서 흔들리지 않기)

**한다:**
- 핫시트 릴레이 (한 브라우저에서 턴 넘기기)
- 목각인형 디오라마 스타일 고정
- Story Bible 기반 일관성
- 이미지 컷 릴 (기본), Veo는 옵션 버튼

**안 한다:**
- 로그인/계정, 실시간 멀티유저 동기화, 방 만들기
- 영상 실시간 생성에 데모 걸기
- DB (state는 메모리/localStorage면 충분)
- 모바일 대응, 공유 기능, 투표 기능 — 발표에서 "로드맵"으로 말만 한다

## 장애 대응

- 이미지 생성 실패 → 재시도 1회 → 그래도 실패하면 해당 컷은 텍스트 카드(내레이션 카드)로 렌더 (연출로 보임)
- 시작 화면에서 키 누락/바이블 생성 실패 → "데모 모드로 체험" 진입 제안 (DEMO 배지 표시). **릴레이 도중 씬 생성 실패는 목업 대체 금지** — 콘셉트 에러 + 재시도 버튼 (입력 보존)
- 새로고침 복구는 텍스트만 복구된다 (이미지는 텍스트 카드로 대체) — 데모/녹화 중 새로고침 금지
- API 키/쿼터 문제 → 데모 영상은 잘 도는 시점에 미리 한 번 찍어둔다 (90분 시점에 1차 녹화 권장)
- 브라우저 키는 임시·제한된 행사 키만 사용하고 localhost 외부에 노출하지 않으며 종료 후 폐기
- 보안 스캔 주의: 인증 없는 HTTP 포트 개방, 외부 개방 DB, PHP 사용 → 시스템이 감지해서 프로젝트 정지될 수 있음
